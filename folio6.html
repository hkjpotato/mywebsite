My thoughts on ReactJS



1st Post: React & Information Visualization

I am thinking what I should put on this folio. I still have several projects I wish to show, but I am really attracted by somthing else now, which is called ReactJS. Thus at this moment I will just share some of my thoughts about React while learning it (like a post in a blogger).


So I am still a new hand in this fantastic front end world. I got most of my front end experience from the Ford Remote Control Car project. And I feel lucky to have a chance to work in the real industry as a UI developer now. More luckily, the company I work, Cvent.Inc, is moving to ReactJS from the traditional MVC mdoel developed by ASP.NET. As many beginners, I start to learn React from the official website, some egghead videos, etc. And then I heard that Uber Visualization Team is now using ReactJS to do the Visualization.


Wait...What?!


What I know about React at this moment, is that it uses one way data flow, views each component as a state machine. And because of the virtual dom, it is fast. But to do visualization? to manipulate SVG? It sounds weird for me at first.


For people who have experience with web visualization, chances are we are using a library called D3. And for a period of time, I thought that D3 is the only choice to manipulate SVG and build those charts and graphs. Not until I read this great article from Jerome Cukier "You may not d3", did I realized that many functions of D3 can be done with native Javascript. So, even without D3, we can make the visualization using React.


The first question comes to me for using React is to do vis is how to realize "brushing and linking" in multi-views interface. "Brushing and linking" means interactive changes made in one visualization are automatically reflected in the other visualizations. For example, hovering on the data point in one view will also highlight the corresponding point in another view.


The key problem here is how to build a connection between different views. In the uber visualization project, I solved this problem in the following way (for bar chart).

  1. Bind the data to the svg element.
  2. Attach a callback function to hear the action on that element (e.g. click& hover)
  3. In the callback, check the attached data and use it as a parameter to modified the other view

However, in React, I am thinking where I should put this callback function. Should I call it from the child element, or from the parent element. Does a change in UI, like css style, necessarily means a change in "state"? If not, who should control this change? How I should encapsulate each view so as to make them reusable?


The best way to learn something is to actually work on it. To get myself be familar with React, I will work on the following steps.


  1. Build a vis based on React
  2. Change the UI with/without changing the state
  3. Link two vis

2nd Post: React Animation

Since I have to be prepared for the coming React work I am now more focused on understanding the basic of this library instead of Info Vis. And then I find this amazing video


"Animation APIs parameterized by eg duration and curve are fundamentally opposed to continuous, fluid interactivity" -- Andy Matuschak


To sum up, animation in the react is:

  • Interruptible
  • Concurrent
  • Composable
  • Last but not least: Declarative

This is amazing for me since most of my animations are based on CSS3, e.g. the icon loading animation for my website and cube rotation app. After all, there are all css transition and animation. I remember that animations in AngularJS are also based on CSS classes. And while it is easy to use, it is still a blackbox for the developers.
